# Codex 工程协作指南

## 1. 采用的官方工作原则

依据 OpenAI Codex 官方手册，本项目采用以下协作方式：

- 用清晰提示词描述目标、上下文、约束和完成标准。
- 对复杂任务先计划，再实现。
- 用 `AGENTS.md` 固化项目规则、运行命令、工程约定、审查要求和完成定义。
- 通过测试、检查和代码评审闭环提升可靠性。
- 用技能和 MCP 沉淀可复用流程，例如读取 PDF、查询官方文档、控制浏览器、访问 GitHub 或 Figma。
- 保持安全边界，理解沙箱、审批和网络访问对本地工作的影响。

官方参考：<https://developers.openai.com/codex/codex-manual.md>

## 2. Codex 提示词模板

```text
目标：
开发/修改/审查 [功能或文档]，关联需求 [FR-xxx/NFR-xxx]。

上下文：
- 相关文件：
- 相关接口：
- 相关风险：
- 现场约束：

约束：
- 不使用真实患者数据。
- 高风险操作要审计。
- 遵守 AGENTS.md 和 docs/prelaunch 中的编号规则。
- 若影响安全、接口或上线项，更新对应文档。

完成标准：
- 实现需求并通过相关测试。
- 更新追踪矩阵。
- 给出验证结果和残余风险。
```

## 4. Codex 适用任务

| 任务 | 做法 |
| --- | --- |
| 需求拆解 | 让 Codex 按 PRD/SRS 编号生成需求、验收和测试 |
| 接口设计 | 提供医院接口样例，让 Codex 生成 OpenAPI、字段映射和 Mock |
| 设备适配 | 让 Codex 为每类设备建立适配器接口和模拟器 |
| 测试生成 | 按 FR/RISK 编号生成单元、集成、E2E、安全和性能测试 |
| 文档维护 | 每次功能变更同步更新 SRS、测试、风险和追踪矩阵 |
| 代码评审 | 重点检查权限、审计、隐私、错误处理、并发和数据一致性 |

## 5. Codex 安全使用要求

- 不把真实患者数据、生产密钥、医院 VPN 信息或未脱敏日志发给模型。
- 需要联网查询时优先使用官方来源，并把结论写入参考文档。
- 涉及删除、迁移、批量改动、生产环境、密钥和外部系统写操作时必须人工确认。
- 生成代码后必须运行相关测试；不能运行时说明原因和替代检查。
- 对医疗、合规、安全结论保持“建议”和“待专业确认”的边界。

## 6. 评审清单

Codex 或人工评审每次至少关注：

- 是否违反预期用途，暗示诊断或治疗。
- 是否可能泄露患者信息或真实视频。
- 权限是否覆盖 UI、API 和后台任务。
- 删除、导出、会议外部访问是否有审计和审批。
- 录制文件是否有完整性、容量和失败处理。
- 场景应用和路由切换是否幂等、可回滚。
- 接口失败时是否有降级方案。
- 测试是否覆盖需求和风险编号。

## 7. 后续可创建的技能

项目成熟后建议创建仓库级 Codex skills：

| 技能 | 用途 |
| --- | --- |
| `or-requirements-sync` | 从 PRD/SRS 自动检查编号、追踪矩阵和测试覆盖 |
| `or-interface-mapper` | 根据医院接口文档生成字段映射和 Mock |
| `or-risk-review` | 检查变更是否影响风险和合规材料 |
| `or-release-gate` | 正式开放前自动检查发布清单 |

这些技能应放在仓库 `.agents/skills/` 下，或在需要分发时打包为 Codex 插件。
