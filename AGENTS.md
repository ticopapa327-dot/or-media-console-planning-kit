# OR Media Console 项目协作约定

## 项目定位

本仓库用于规划和开发同类型的数字一体化手术室媒体控制系统。当前需求基线来自已脱敏的标准版说明资料以及公开同类系统资料。原始 PDF 不随公开仓库发布。

系统的默认预期用途是手术室媒体控制、示教、会议、存档和文档管理。除非完成新的法规论证和注册路径评估，不得把录制图像、视频或算法输出声明为诊断或治疗依据。

## 工作方式

所有功能、测试、风险和上线项必须使用稳定编号：

- 功能需求：`FR-001`
- 非功能需求：`NFR-001`
- 用例：`UC-001`
- 接口：`INT-001`
- 风险：`RISK-001`
- 测试：`TC-001`
- 上线检查项：`REL-001`

进行 Codex 任务时，提示词应包含目标、上下文、约束和完成标准。复杂任务先形成计划，再进入实现。完成定义应至少包括：需求编号更新、相关测试或检查通过、风险和文档影响已评估。

## 医疗与隐私约束

- 不得在样例、测试、日志、截图或提交信息中使用真实患者信息。
- 当前 MVP 暂不支持从 HIS/EMR 获取患者信息；患者和手术信息使用手动录入或合成假数据替代。
- 患者标识、手术视频、图片、会议记录和操作日志均按敏感医疗数据处理。
- 默认采用最小权限、审计留痕、加密传输、加密存储、数据保留期限和可追溯删除策略。
- 与 HIS、EMR、PACS、LDAP、对象存储、会议平台和设备控制器集成时，优先使用标准协议或医院批准的接口。

## 工程规范

- 先读现有文档和代码，再改动。
- 文档更新放在 `docs/`，正式开放前文档放在 `docs/prelaunch/`。
- 用户可见流程必须有验收标准；安全、权限、存储、审计、告警相关改动必须同步更新测试和风险。
- 重要架构决策应补充 ADR 或写入架构文档的“决策记录”。
- 竞品或法规事实要引用官方来源；无法确认时标注为“待核验”。

## 验证要求

代码项目建立后，应在仓库内补充实际命令。当前文档阶段的检查重点：

- 文档编号无重复。
- 需求、风险、测试、上线检查项存在基本追踪关系。
- 任何医疗用途、数据出境、云部署、AI 处理、真实患者数据使用均有合规评估记录。
